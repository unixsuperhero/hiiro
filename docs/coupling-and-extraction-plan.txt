вҸә Take a look at both. Key differences:

  sqlite3 gem вҖ” low-level:
  db = SQLite3::Database.new("hiiro.db")
  db.execute("SELECT * FROM todos WHERE status = ?", ["active"])
  # => [["abc123", "fix bug", "not_started", ...]]  (plain arrays)

  sequel gem вҖ” higher-level ORM:
  DB = Sequel.sqlite("hiiro.db")
  DB[:todos].where(status: "active").all
  # => [{id: "abc123", text: "fix bug", status: "not_started", ...}]  (hashes)

  Sequel pros for this project: named-column hash results (not positional
  arrays), chainable queries, schema migrations built-in, no raw SQL for common
  ops. The sqlite3 gem is its underlying driver anyway.

  Given that we have ~15 tables and complex query patterns (filters, sorts,
  joins for tasks/branches/PRs), Sequel would meaningfully reduce boilerplate.
  But it's a heavier dependency.

  вҳ… Insight в”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җ
  The HIIRO_INVOCATION_ID pattern is the cleanest way to link dispatch hops
  across exec() boundaries вҖ” bin/h generates an ID and exports it as an env var
  before exec'ing bin/h-branch, which reads it to attach its resolution to the
  same parent record. No IPC needed, just a shared env var flowing through the
  process tree.
  в”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җ

  My recommendation for the resolution design:

  // Invocation вҖ” one per shell command entered
  { id: "inv_abc", command: "h branch saved", pwd: "/Users/josh/work/oncall",
    task: "oncall", worktree: "oncall/main", git_branch: "sa-form-...",
  entered_at: "..." }

  // InvocationResolution вҖ” N per invocation, ordered by depth
  { invocation_id: "inv_abc", depth: 0, bin: "h",        subcmd: "branch", args:
   ["saved"] }
  { invocation_id: "inv_abc", depth: 1, bin: "h-branch", subcmd: "saved",  args:
   []        }

  Why this wins over single-record: you get the human-readable command (h branch
   saved) for display/search, AND the precise execution context for debugging.
  depth: 0 always reflects what the user typed.

вқҜ what did we decide on for migration...do we dual write?                       

вҸә Yes вҖ” dual-write. Every mutation writes to SQLite first (primary store), then
  rewrites the original YAML file as a backup. The YAML files stay live and in
  sync, not just renamed.

  The one-time auto-migration on first run (when hiiro.db doesn't exist yet)
  goes the other direction: it reads the existing YAMLs вҶ’ imports into SQLite вҶ’
  renames each source to .yml.bak. After that, all subsequent writes go
  DB-primary + YAML-backup.

  So the flow is:

  First run (no hiiro.db):
    YAMLs вҶ’ [import] вҶ’ hiiro.db
    YAMLs вҶ’ renamed to .yml.bak

  Every run after:
    Reads  вҶҗ hiiro.db (primary)
    Writes вҶ’ hiiro.db + .yml (dual-write backup)

  The .yml backups let you roll back to YAML-only if something breaks, without
  losing any data written after migration.

     Option C: Hiiro::Effects Injectable Interface

     Context

     Hiiro's test suite is hard to maintain because side effects (tmux, git,
     filesystem, process exec) are
     interleaved with domain and presentation logic in bin files and manager
     classes. Claude has repeatedly
     mixed these concerns, making tests brittle or impossible to write.

     The good news: Hiiro::Tmux and Hiiro::Git already have an internal adapter
     pattern вҖ” they route
     all calls through three private methods: run_safe (backtick), run_system
     (system), run_success?
     (silent system). The gap is that these aren't injectable, so tests can't
     substitute them.

     Goal: Make side effects injectable via a Hiiro::Effects namespace. Real
     code gets real
     implementations; tests get lightweight doubles. No new abstraction layers вҖ”
      just expose and inject
     what already exists.

     Not a big-bang rewrite. This is incremental: start with Tmux and Git
     (highest test value), then
     add Filesystem and Process. Each step is independently useful.

     Step 1: Define the Effects interfaces (lib/hiiro/effects.rb)

     Three interfaces, each with a real implementation and a test double:

     module Hiiro
       module Effects
         # Wraps backtick + system calls for tmux and git
         class Executor
           def capture(*args)  = `#{args.shelljoin} 2>/dev/null`
           def run(*args)      = system(*args)
           def check(*args)    = system(*args, out: File::NULL, err: File::NULL)
         end

         # Test double вҖ” records calls, returns configured stubs
         class NullExecutor
           attr_reader :calls

           def initialize
             @calls   = []
             @stubs   = {}
           end

           def stub(pattern, output)
             @stubs[pattern] = output
           end

           def capture(*args)
             @calls << { method: :capture, args: args }
             find_stub(args) || ""
           end

           def run(*args)
             @calls << { method: :run, args: args }
             find_stub(args).nil? ? true : find_stub(args)
           end

           def check(*args)
             @calls << { method: :check, args: args }
             find_stub(args).nil? ? true : !!find_stub(args)
           end

           def called?(pattern)
             @calls.any? { |c| c[:args].any? { |a| a.to_s.include?(pattern.to_s)
      } }
           end

           private

           def find_stub(args)
             key = args.map(&:to_s).join(' ')
             @stubs.find { |pat, _| key.include?(pat.to_s) }&.last
           end
         end

         # Filesystem effects (read/write/delete)
         class Filesystem
           def write(path, content)    = (FileUtils.mkdir_p(File.dirname(path));
      File.write(path, content))
           def read(path)              = File.read(path)
           def exist?(path)            = File.exist?(path)
           def mkdir_p(path)           = FileUtils.mkdir_p(path)
           def rm(path)                = File.delete(path) if File.exist?(path)
           def mv(src, dst)            = FileUtils.mv(src, dst)
           def glob(pattern)           = Dir.glob(pattern)
         end

         class NullFilesystem
           attr_reader :writes, :deletes

           def initialize
             @store   = {}
             @writes  = []
             @deletes = []
           end

           def write(path, content)    = @writes  << path and @store[path] =
     content
           def read(path)              = @store[path] || raise(Errno::ENOENT,
     path)
           def exist?(path)            = @store.key?(path)
           def mkdir_p(path)           = nil
           def rm(path)                = @deletes << path and
     @store.delete(path)
           def mv(src, dst)            = @store[dst] = @store.delete(src)
           def glob(pattern)           = @store.keys.select { |k|
     File.fnmatch(pattern, k) }
         end
       end
     end
